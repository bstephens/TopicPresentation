Local Referencing Environments

 

Local Variables

  • Scope of local variables is usually the body of the subprogram in which they are defined
  • Can be either stack dynamic or static
    • Stack dynamic
      • Bound to storage when the subprogram begins execution and are unbound when that execution terminates
      • Provides flexibility to the subprogram and essential for recursive subprograms
      • Storage for local variables in an active program can be shared with the local variables in all inactive subprograms (useful for small memory computers)
      • Cost of the time required to allocate, initialize, and deallocate for each call to the subprogram
      • Access to stack dynamic local variables must be indirect while access to static local variables can be direct
    • Static
      • Slightly more efficient because they require no run-time overhead for allocation and deallocation
      • Direct access to local variables is more efficient than indirect access
      • Allow subprograms to be history sensitive
      • Unable to support recursion
  • Contemporary languages default local variables to stack dynamic
    • C and C++ allow locals to be declared static
    • Ada subprograms and the methods of C++, Java, and C# have only stack dynamic local variables
    • Only variables with restricted scope are declared in Lua
      • Declared using the local declaration
      • Nondeclared variables are global
      • Access to local variables in Lua is faster than access to global variables according to Ierusalimschy (2006)

Nested Subprograms

  • Nesting subprograms originated with Algol 60 to create a hierarchy of both logic and scopes
  • All of the direct descendants of C do not allow nesting of subprograms
  • JavaScript, Python, Ruby, and Lua allow subprogram nesting
